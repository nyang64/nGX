  agentmail's implementation:                         
  
  Inbound Flow                                                                                                                                        
  MX record → SES Receipt Rule → S3 (raw .eml) + SNS/SQS
                                          ↓                                                                                        
                                    Lambda / ECS worker
                                    (parse MIME, thread, store)                                                                    
                                          ↓                                                                                      
                                     Database + Svix
                                                                                                                                   
  SES receipt rules let you do two things simultaneously: drop the raw .eml into S3, and fire an SNS notification. The processing  
  worker reads the raw email from S3, parses it (MIME decoding, extracting text/HTML, stripping quoted history for extracted_text),
   reconstructs threads from In-Reply-To/References headers, then writes structured records to the database and triggers Svix for webhook delivery.                                                                                                              

  The extracted_text/extracted_html fields tell us they run a reply-stripping step — likely something like email-reply-parser or a 
  custom heuristic — to give agents just the new content.
                                     
  Outbound Flow

  POST /inboxes/{id}/messages/send
          ↓
    API service validates, writes record (status=queued)
          ↓                                                                                                                        
    SES SendRawEmail (builds MIME, sets In-Reply-To/References for threading)
          ↓                                                                                                                        
    SES delivery event (SNS) → update record status (sent/bounced/etc.)                                                          
                                                                                                                                   
  Outbound is simpler. They construct the MIME message, call SES, and get back an SES MessageId. They likely store both their      
  internal message_id and the SES MessageId for bounce/complaint correlation via SES event notifications.                          
                                                                                                                                                                             
  Storage Layer (the interesting part)
                                         
  best guess: PostgreSQL (Aurora Serverless) + S3

  text (full body), html (full body),                                                                                            
  extracted_text, extracted_html,                                                                                                  
  in_reply_to, references[],                                                                                                       
  headers (jsonb),          
  labels (text[]),

  Key storage decisions:                                                                                                           
                                                                                                                                   
  Full text/html body in the DB, not S3. They return full body in the API response without a separate fetch step, so it's almost   
  certainly in the DB. PostgreSQL handles typical email body sizes fine. The raw .eml in S3 is an archive/source-of-truth, not the 
  serving path.                                                                                                                    
                                                                                                                                 
  extracted_text is derived, stored. They likely compute it once at ingest time and store it. Recomputing on every read would be   
  wasteful and non-deterministic.
                                                                                                                                   
  Thread reconstruction happens at ingest by walking In-Reply-To → References chain. If no match, a new thread is created. Subject 
  normalization (strip Re:, Fwd:) is likely a fallback.
                                                                                                                                   
  Attachments are stored in S3, metadata in DB. The API returns attachment metadata (id, filename, size, content_type) in the      
  message object, with a separate endpoint to fetch content — classic S3-presigned-URL or proxy pattern.

  Svix Integration
                  
  Svix is essentially a managed webhook infrastructure SaaS. Their integration is likely:
                                                                                                                                   
  svix_client.message.create(app_id=inbox_id, MessageIn(
      event_type="message.received",                                                                                               
      payload={ "type": "event", "event_type": "...", "message": {...}, "thread": {...} }                                        
  ))                                                                                                                               
                                                                                                                                 
  Each inbox maps to a Svix "application." Webhook endpoints registered by users are Svix "endpoints." This is why they get retry  
  logic, signing (HMAC via svix-signature header), delivery logs, and a dashboard for free — it's all Svix.                      
                                                                                                                                   
  The HMAC secret the user configures is just the Svix endpoint signing secret.

  The fundamental architectural difference: they outsource the hard, stateful distributed systems problems (email transport,       
  webhook reliability) to SES and Svix, reducing what they own to: inbox management, MIME parsing, thread reconstruction, and the
  API layer. Their core engineering problem is much smaller than ours.                                                             
                                                                                                                                 
  What we built is more flexible — Kafka at the center gives you fan-out to multiple consumers (search indexing, embeddings,       
  analytics) without coupling. Their SNS-triggered Lambda approach works well but adding a new consumer means a new SNS
  subscription and Lambda, which is more operational overhead per consumer.                                                        
                                                                                                                                 
  The search and embedder services we have are a good example — with Kafka, the email-pipeline emits once and both the search      
  indexer and embedder consume independently. In their architecture those would require separate SNS subscriptions or a downstream
  fan-out step. 


  ///// partition strategies in postgres

  1. Range Partitioning (most common for time-series / messages)
                                                                                                                                   
  -- Parent table                                                                                                                
  CREATE TABLE messages (                                                                                                          
      message_id      UUID NOT NULL,                                                                                             
      inbox_id        TEXT NOT NULL,
      thread_id       TEXT,                                                                                                        
      subject         TEXT,
      body_text       TEXT,                                                                                                        
      created_at      TIMESTAMPTZ NOT NULL                                                                                       
  ) PARTITION BY RANGE (created_at);                                                                                               
   
  -- Monthly partitions                                                                                                            
  CREATE TABLE messages_2025_01 PARTITION OF messages                                                                            
      FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');                                                                            
   
  CREATE TABLE messages_2025_02 PARTITION OF messages                                                                              
      FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');                                                                          
                                                                                                                                   
  CREATE TABLE messages_2025_03 PARTITION OF messages
      FOR VALUES FROM ('2025-03-01') TO ('2025-04-01');                                                                            
                                                                                                                                 
  -- Default partition catches anything that doesn't match                                                                         
  CREATE TABLE messages_default PARTITION OF messages DEFAULT;
                                                                                                                                   
  ---                                                                                                                            
  2. List Partitioning (partition by discrete values)
                                                                                                                                   
  CREATE TABLE messages (
      message_id  UUID NOT NULL,                                                                                                   
      inbox_id    TEXT NOT NULL,                                                                                                 
      region      TEXT NOT NULL,
      body_text   TEXT,
      created_at  TIMESTAMPTZ NOT NULL                                                                                             
  ) PARTITION BY LIST (region);
                                                                                                                                   
  CREATE TABLE messages_us_east PARTITION OF messages                                                                            
      FOR VALUES IN ('us-east-1', 'us-east-2');
                                                                                                                                   
  CREATE TABLE messages_us_west PARTITION OF messages
      FOR VALUES IN ('us-west-1', 'us-west-2');                                                                                    
                                                                                                                                 
  CREATE TABLE messages_eu PARTITION OF messages
      FOR VALUES IN ('eu-west-1', 'eu-central-1');
                                                                                                                                   
  ---
  3. Hash Partitioning (even distribution, good for inbox_id)                                                                      
                                                                                                                                   
  CREATE TABLE messages (
      message_id  UUID NOT NULL,                                                                                                   
      inbox_id    TEXT NOT NULL,                                                                                                 
      body_text   TEXT,
      created_at  TIMESTAMPTZ NOT NULL
  ) PARTITION BY HASH (inbox_id);                                                                                                  
   
  -- 8 partitions, evenly distributed by hash of inbox_id                                                                          
  CREATE TABLE messages_p0 PARTITION OF messages                                                                                 
      FOR VALUES WITH (MODULUS 8, REMAINDER 0);                                                                                    
                                                                                                                                   
  CREATE TABLE messages_p1 PARTITION OF messages
      FOR VALUES WITH (MODULUS 8, REMAINDER 1);                                                                                    
                                                                                                                                 
  CREATE TABLE messages_p2 PARTITION OF messages
      FOR VALUES WITH (MODULUS 8, REMAINDER 2);
                                                                                                                                   
  -- ... through p7
  CREATE TABLE messages_p7 PARTITION OF messages                                                                                   
      FOR VALUES WITH (MODULUS 8, REMAINDER 7);                                                                                  

  ---                                                                                                                              
  4. Composite Partitioning (sub-partitioning)
                                                                                                                                   
  Partition by range, then sub-partition by hash — good for messages table:                                                      
                                                                                                                                   
  CREATE TABLE messages (
      message_id  UUID NOT NULL,                                                                                                   
      inbox_id    TEXT NOT NULL,                                                                                                 
      body_text   TEXT,                                                                                                            
      created_at  TIMESTAMPTZ NOT NULL
  ) PARTITION BY RANGE (created_at);                                                                                               
                                                                                                                                 
  -- Monthly range partition, then sub-partitioned by hash of inbox_id                                                             
  CREATE TABLE messages_2025_01 PARTITION OF messages
      FOR VALUES FROM ('2025-01-01') TO ('2025-02-01')                                                                             
      PARTITION BY HASH (inbox_id);                                                                                                
   
  CREATE TABLE messages_2025_01_p0 PARTITION OF messages_2025_01                                                                   
      FOR VALUES WITH (MODULUS 4, REMAINDER 0);                                                                                  
                                                                                                                                   
  CREATE TABLE messages_2025_01_p1 PARTITION OF messages_2025_01
      FOR VALUES WITH (MODULUS 4, REMAINDER 1);                                                                                    
                                                                                                                                 
  CREATE TABLE messages_2025_01_p2 PARTITION OF messages_2025_01                                                                   
      FOR VALUES WITH (MODULUS 4, REMAINDER 2);
                                                                                                                                   
  CREATE TABLE messages_2025_01_p3 PARTITION OF messages_2025_01                                                                 
      FOR VALUES WITH (MODULUS 4, REMAINDER 3);
                                                                                                                                   
  ---
  Indexes on Partitioned Tables                                                                                                    
                                                                                                                                 
  -- Index created on parent automatically propagates to all partitions
  CREATE INDEX idx_messages_inbox_id ON messages (inbox_id);                                                                       
  CREATE INDEX idx_messages_created_at ON messages (created_at);                                                                   
                                                                                                                                   
  -- Composite index for the common query pattern                                                                                  
  CREATE INDEX idx_messages_inbox_created ON messages (inbox_id, created_at DESC);                                               
                                                                                                                                   
  ---
  Dropping Old Partitions (key advantage of range partitioning)                                                                    
                                                                                                                                   
  -- Instantly drop a month of old data — no DELETE, no vacuum needed
  DROP TABLE messages_2024_01;                                                                                                     
                                                                                                                                 
  -- Or detach and archive first                                                                                                   
  ALTER TABLE messages DETACH PARTITION messages_2024_01;                                                                        
                                                                                                                                   
  This is the main operational reason to use range partitioning over hash — old partitions can be dropped in milliseconds with zero
   impact on the rest of the table.
                                                                                                                                   
  ---                                                                                                                            
  For agentmail's messages table
                                                                                                                                   
  The best strategy is range by created_at (monthly) — gives you:
  - Fast time-range queries (fetch messages from last 30 days hits 1-2 partitions)                                                 
  - Easy archival by dropping old partitions                                                                                       
  - Predictable partition sizes as volume grows                                                                                    
                                                                                                                                   
  Hash by inbox_id is useful if your primary query pattern is always scoped to a single inbox, but range on time is more           
  universally beneficial for email.

/// vector search performance and linearage info
  Milvus                                                                                                                         

  - Built by Zilliz (a Chinese-American startup), open-sourced 2019                                                                
  - Written in Go + C++
  - Roots in academic ANN research, particularly Facebook's Faiss library (which Milvus uses internally for the actual index       
  computation)                                                                                                                     
  - Architecture inspired by cloud-native data systems (Kafka for WAL, MinIO/S3 for storage, etcd for metadata)
  - Faiss itself is a C++ library from Facebook AI Research — the gold standard for ANN algorithms                                 
                                                                                                                                   
  Qdrant                                                                                                                           
                                                                                                                                   
  - Built in Rust, open-sourced 2021 by a Berlin-based startup                                                                     
  - No legacy lineage — built from scratch with Rust's memory safety and performance
  - Uses HNSW as primary index, with scalar and product quantization added later                                                   
  - Closest spiritual cousin is Elasticsearch in terms of API design philosophy, but no shared code                                
                                                                                                                                   
  Weaviate                                                                                                                         
                                                                                                                                   
  - Built in Go, open-sourced 2019 by a Dutch startup (SeMI Technologies)                                                          
  - Roots in semantic search and knowledge graphs — originally positioned as a "semantic search engine" before the vector DB
  category existed                                                                                                                 
  - Uses HNSW internally, licensed from the academic HNSW paper implementation                                                   
  - Has a GraphQL API — reflects its knowledge graph origins                                                                       
                                                                                                                                   
  Pinecone                                                                                                                         
                                                                                                                                   
  - Fully managed, closed source, founded 2019                                                                                     
  - Built by ex-AWS engineers (the team behind Amazon SageMaker)
  - Architecture not publicly disclosed in detail                                                                                  
  - Likely uses a hybrid of IVF+PQ internally given their scale claims                                                             
  - No open-source lineage

  The Actual Family Tree                                                                                                         
                                                                                                                                   
  Academic ANN research (2010s)
  │                                                                                                                                
  ├── Facebook Faiss (2017, C++ library)                                                                                         
  │         ↓                                                                                                                      
  │       Milvus (uses Faiss internally)                                                                                           
  │       Pinecone (likely uses Faiss concepts)
  │                                                                                                                                
  ├── HNSW paper — Malkov & Yashunin (2016)                                                                                      
  │         ↓                                                                                                                      
  │       Qdrant (own Rust HNSW implementation)                                                                                  
  │       Weaviate (own Go HNSW implementation)                                                                                    
  │       pgvector (own C HNSW implementation)                                                                                   
  │                                                                                                                                
  └── Elasticsearch / Lucene (text search roots)
            ↓                                                                                                                      
          OpenSearch (with k-NN plugin using Faiss/NMSLIB)                                                                       
          Elasticsearch 8.x (with native vector search)                                                                            
   
  ---                                                                                                                              
  The Postgres-Native Vector Story                                                                                               
                                                                                                                                   
  The only serious vector search with actual Postgres roots is pgvector — a Postgres extension written in C that plugs directly
  into Postgres's index infrastructure. Everything else is a separate system that you query via its own API and sync data into from
   your primary DB.                                                                                                              
                                                                                                                                   
  There's also pgvectorscale from Timescale which adds a new index type called StreamingDiskANN — a disk-based ANN index that      
  doesn't require full RAM residency, extending pgvector's practical range to ~100-500M vectors. Still Postgres-native, but a
  separate extension on top of pgvector.                                                                                           
                                                                                                                                 
  ---
  Practical Takeaway
                                                                                                                                   
  If you want to stay in the Postgres world:
  <50M vectors:   pgvector + HNSW — fully native                                                                                   
  50M-500M:       pgvector + pgvectorscale (StreamingDiskANN)                                                                    
  500M+:          Leave Postgres, use Qdrant or Milvus                                                                             
                                                                                                                                   
  If you're willing to run a separate system, Qdrant is the current community favorite for self-hosted — Rust performance, good    
  compression, simple operationally, no JVM, active development.

  Key Design Details                                                                                                               
                                                                                                                                   
  ┌────────────────────────┬────────────────────────────────────────────────────────────────────────┐                            
  │        Concern         │                                 Detail                                 │
  ├────────────────────────┼────────────────────────────────────────────────────────────────────────┤
  │ Redis channel name     │ ws:events:{orgID} — per-tenant scoping                                 │
  ├────────────────────────┼────────────────────────────────────────────────────────────────────────┤
  │ Subscription lifecycle │ Lazy: created on first WS client, destroyed on last disconnect         │                              
  ├────────────────────────┼────────────────────────────────────────────────────────────────────────┤                              
  │ Slow client handling   │ Non-blocking send — slow clients are dropped, not backpressured        │                              
  ├────────────────────────┼────────────────────────────────────────────────────────────────────────┤                              
  │ Webhook ordering       │ Keyed by webhook ID in Kafka, so deliveries per webhook are sequential │                            
  ├────────────────────────┼────────────────────────────────────────────────────────────────────────┤                              
  │ Webhook auth           │ HMAC-SHA256 signature in X-nGX-Signature header                  │                            
  ├────────────────────────┼────────────────────────────────────────────────────────────────────────┤                              
  │ Delivery record        │ Every webhook attempt is persisted (pending → success/failed)          │                            
  ├────────────────────────┼────────────────────────────────────────────────────────────────────────┤                              
  │ Event isolation        │ Both flows are org-scoped — events never cross tenant boundaries       │                            
  └────────────────────────┴────────────────────────────────────────────────────────────────────────┘                              
                                                                                                                                 
  The architectural split is clean: Redis pub/sub is purely real-time fan-out to open WebSocket connections (stateless,            
  fire-and-forget), while webhooks go through a second Kafka queue to get persistence, retryability, and per-webhook ordering.
